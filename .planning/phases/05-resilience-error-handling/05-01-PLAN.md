    # ADR-0005: Resilience & Error-Handling Invariants

    ## Status
    ACCEPTED (2026-06-21)

    ## Context
    Phase 5 layers adversarial-condition handling onto the Phase 2/3/4 pipeline so every failure mode produces a clear German error string returned from RefreshAccount / InitializeSession2, never a Lua error, never a partial result, never a silently-advanced `since` watermark. Six requirements (ERR-01..06) per .planning/REQUIREMENTS.md drive the design. The decision is split into six invariants (one per requirement) plus two carve-outs (SSL handshake bypass, HTTP-date fallback) plus a sleep-mechanism choice.

    ## Decision

    ### Invariant 1: ERR-01 token-mint invalid_grant → LoginFailed (D-61)
    Already shipped in Phase 2 (src/errors.lua D-24 dispatch: 400 → LoginFailed). Plan 05-02 only extends spec/auth_spec.lua with an explicit fixture-driven ERR-01 test using the existing spec/fixtures/auth/auth_invalid_grant.json fixture. No source change. The LoginFailed constant is reserved for token-mint failures — distinguished from ERR-04 (post-mint 401) which surfaces error.token_revoked.

    ### Invariant 2: ERR-02 5xx retry-with-backoff (D-62)
    Inside M_http.get_json / post_form: iterate `for attempt = 1, 3 do`. Sleep durations: `{1, 2, 4}` seconds (exponential base 2). On exhausted retries, the surfaced status is whatever the body-shape inference yielded (typically `nil` → routes to error.network via M_errors.from_http_status — see Invariant 5). Each attempt emits exactly ONE INFO log line via M_log.info with format `"HTTP retry: attempt=N/3 status=NNN url=URL after_ms=NNNN"` (Bearer redacted by M_log.redact + structurally absent because headers are never concatenated into the log string). Recursive retry is REJECTED (Pitfall §2: blows Lua 200-frame call stack).

    ### Invariant 3: ERR-03 429 rate-limiting (D-63)
    Single retry per endpoint. Honor Retry-After header (integer-seconds only — see Carve-out 2). Cap at 60s upper bound (stays within MoneyMoney per-call timeout estimate ~30-60s). Default to 30s when Retry-After absent or unparseable. Check both `Retry-After` and `retry-after` header keys (Pitfall §6: server middleware varies on casing). After single retry, if still 429, surface error.rate_limit via M_errors.from_http_status.

    ### Invariant 4: ERR-04 post-mint 401 → error.token_revoked IMMEDIATELY (D-64 COLLAPSED)
    CONTEXT D-64 originally said "perform exactly ONE silent token re-mint". RESEARCH §Pattern-2 collapses this: the silent re-mint is INFEASIBLE under the assertion-grant model + SEC-03 / AUTH-05 invariants. Specifically:
    - Assertion-grant has NO refresh token (CLAUDE.md "Token TTL = 7200s; no refresh token").
    - The api_key is ONLY present during InitializeSession2; SEC-03 / AUTH-05 forbid persisting it to LocalStorage or beyond the InitializeSession2 lifetime.
    - The only re-mint path would require caching the api_key in a module-local Lua variable across RefreshAccount calls, which widens the in-memory exposure window from "during InitializeSession2 only" to "for the lifetime of the MoneyMoney process" — incompatible with the SEC-03 wording's spirit.

    Therefore Phase 5 ships the COLLAPSED contract: on a post-mint 401 received from any resource endpoint (Purchase, Finance), M_http returns the 401 status; the resource-endpoint caller (M_purchases.fetch, M_finance.fetch, M_finance.fetch_account_state) translates the 401 → error.token_revoked German string IMMEDIATELY. NOT LoginFailed (that constant is reserved for ERR-01 token-mint invalid_grant per D-61). User sees the German message "Anmeldung verloren — bitte API-Key in MoneyMoney neu eintragen" and manually re-enters the API key via MoneyMoney's account dialog (which re-invokes InitializeSession2 → fresh mint).

    The reinterpretation preserves the user-facing intent of D-64 ("don't return LoginFailed; the credentials are not bad — the session was revoked") while strictly honoring SEC-03 / AUTH-05.

    Future v2 path: if/when Phase 7 (Optional OAuth Authorization-Code flow per ROADMAP) ships, the refresh-token primitive enables genuine silent re-mint and ERR-04 can be revisited.

    ### Invariant 5: ERR-05 network failure → error.network (D-65)
    Phase 2's design REJECTS pcall around Connection:request (Pitfall §3 in Phase-2 RESEARCH; ADR-0003 Q8 bonus finding). Phase 5 inherits this: network failures (DNS, connect-refused, socket timeout) surface as `conn:request` returning empty body, which the existing src/http.lua L130-132 + L102-105 path catches and returns `(nil, nil, raw)` from get_json / post_form. M_errors.from_http_status(nil, "") returns error.network from D-24 case 1. Plan 05-02 adds an explicit regression test queueing `Mocks.push_response({ content = "" })` to gate this path.

    ### Invariant 6: ERR-06 fail-whole-refresh (D-66)
    Phase 4's 16-step RefreshAccount (src/entry.lua L139-435) already implements the invariant: early returns at Steps 1, 2, 4, 7, 8 discard all in-flight state because Lua's lexical scoping cleans up locals automatically (purchases_by_uuid, payments_by_uuid, fees_by_date, transactions all local). RESEARCH §5 audit confirms structural correctness. Plan 05-05 ships the GATING SPEC ONLY (`spec/refresh_fail_whole_spec.lua`) — no source change. The spec queues purchase success + finance 500 retries-exhausted, asserts (a) German error string returned, (b) purchase call happened (captured_requests), (c) no transactions leaked, (d) second refresh with SAME `since` re-runs from scratch and succeeds.

    ## Carve-outs (known limitations)

    ### Carve-out 1: SSL handshake failures bypass ERR-05
    ADR-0003 Q8 bonus finding: pcall around `conn:request` does NOT catch SSL handshake failures — MoneyMoney aborts the Lua chunk and surfaces the failure in its Protokoll panel. ERR-05's German-string contract covers DNS, connect-refused, socket timeouts; SSL handshake failures (expired certs, MITM-detected mismatches) bypass this contract. Mitigation: TLS 1.2+ is enforced by MoneyMoney's Connection() (ADR-0003 Q8); certificate pinning is explicitly OUT OF SCOPE per REQUIREMENTS.md (defer to OS-level cert verification). User-facing remediation: open MoneyMoney's Protokoll panel to diagnose.

    ### Carve-out 2: HTTP-date Retry-After silently degrades to 30s default
    RFC 7231 §7.1.3 allows Retry-After to be either delta-seconds (integer) or HTTP-date. Phase 5 implements integer-only parsing via `tonumber()` with negative-value rejection. HTTP-date format falls through to the 30s default. Rationale: (a) Zettle has never been observed to send HTTP-date Retry-After in any community fixture or documented response; (b) RFC 7231 §7.1.1.1 date parsing requires ~80 LoC for a code path that may never execute; (c) the fallback is safe — a single 30s sleep stays within the per-call timeout estimate. Verify on first 429 observation; if HTTP-date appears, add a one-line gsub or escalate to a dedicated parser.

    ## Sleep mechanism

    Use `MM.sleep(seconds)` (sandbox primitive documented in https://moneymoney.app/api/webbanking/ — "Unterbricht die Ausführung des Skripts für ein paar Sekunden"). The integer-seconds contract is the documented signature; precision below 1 second is undocumented and out of scope (all Phase-5 sleep durations are integer seconds: 1, 2, 4 for 5xx backoff; up to 60 for rate-limit honoring; 30 for rate-limit default).

    **Important correction**: CONTEXT D-67 + Q9 row use the name `MM.os.sleep` — this is INCORRECT. The documented name is `MM.sleep` (under the `MM.*` top-level helpers, not under `MM.os`). The CI test harness already stubs `_G.MM.sleep = function(s) end` as a no-op at spec/helpers/mm_mocks.lua line 233.

    **Q9 sandbox probe**: optional confirmation per RESEARCH §1. Plan 05-01 (this plan) appends a Q9 block to tools/probe.lua so Yves can run it when convenient. The probe checks `type(MM.sleep) == "function"` and measures `MM.sleep(1)` elapsed time, classifying PASS / PRESENT-BUT-NOOP / ABSENT / FAIL. Result is recorded in docs/adr/0003-sandbox-probe-results.md row Q9 (separate file; Plan 05-01 does NOT pre-add the row — Yves adds it after running the probe). If Q9 returns ABSENT, ADR-0005 must be amended to use a busy-wait fallback (`while os.time() < target do end`) — pcall-wrap MM.sleep so a runtime error on a future MM version falls through to no-backoff retry rather than aborting RefreshAccount.

    ## Worst-case timing budget

    Per RESEARCH §9:
    - Single-endpoint 5xx storm: ~9s (1s + 2s backoff + 3× ~500ms-2s HTTP roundtrips).
    - 3-endpoint 5xx storm (purchase + finance state + finance transactions): ~27s.
    - Single 429 with Retry-After: up to 60s (capped).

    The 3-endpoint 5xx worst case fits within MoneyMoney's per-call timeout estimate (~30-60s per ADR-0003 + community survey) but is uncomfortably close. Mitigation strategies (NOT applied in v1.0.0):
    - Reduce backoff curve to `{1, 2}` (saves 4s per endpoint, ~12s total).
    - Reduce MAX_ATTEMPTS to 2.
    - Implement a global RefreshAccount timeout that cancels remaining sleeps once a budget is exceeded.

    v1.0.0 ships the `{1, 2, 4}` 3-attempt curve. Monitor real-world reports; tune in v1.0.x if observed breaches occur.
