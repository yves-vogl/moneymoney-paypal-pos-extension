## Summary

Phase 6 has already done **substantially more** Scorecard-relevant work than CONTEXT 06.1 (drafted 2026-06-17) assumes. The ci.yml split exists, top-level `permissions: read-all` is in place (as a per-job `contents: read`), release.yml exists with three-job least-privilege shape, branch protection on `main` is live with `enforce_admins=true`, signed commits required, linear history, no force-push/delete, three required status checks, and the `setup-branch-protection.sh` scripted runbook is committed. Phase 6 R1 security review (16 findings) catalogues the remaining gaps that 6.1 must close.

Three hard surprises change Phase 6.1's scope materially:

1. **OpenSSF Scorecard does NOT currently detect Semgrep as a SAST tool.** D-33/D-34/SEC-08 cannot deliver SAST=10 with Semgrep alone unless [ossf/scorecard#5103](https://github.com/ossf/scorecard/pull/5103) lands first. The PR was opened 2026-06-23, single commit, awaiting two code-owner reviews. We have three response paths (deliver Semgrep + Snyk dual-track / fall back to a Scorecard-detected tool / accept SAST=N/A with documented rationale) — recommendation in §5.
2. **Scorecard Branch-Protection ≥ 8 requires `required_approving_review_count ≥ 1`.** Current state is `0` (per `tools/setup-branch-protection.sh` line 70) — that caps Branch-Protection at Tier 1 = **3/10**, not the ≥8 acceptance bar. CONTEXT D-31 acknowledged the solo-maintainer trade-off but did not quantify the score impact. Decision needed: either accept a lower aggregate target or introduce a bypass mechanism (separate-session self-review, bot reviewer).
3. **`ADR-0004` is already taken** (`0004-finance-api-scope-and-fee-fallback.md`, Phase 4). Next-free ADR number is **`0009`**. CONTEXT D-36 / DOC-N must rename to `0009-openssf-scorecard-stance.md`.

The remainder of CONTEXT's plan stands. SHA pinning, Token-Permissions hardening, `persist-credentials: false` fixes, CII passing badge, and the ADR + SECURITY.md prose are mechanical, low-risk, and well-supported by current tooling.

**Primary recommendation:** Sequence the work as **(1) SHA-pin all `uses:` references + close 4 Dependabot PRs + persist-credentials hardening as one combined PR, (2) Semgrep workflow + ADR-0009 disclosing the SAST scoring caveat, (3) Branch-protection script update + SCORECARD_READ_TOKEN runbook + the explicit accept-or-revise call on the 0-reviewer trade-off, (4) CII badge + SECURITY.md supply-chain section as the closing docs PR.** Realistic Scorecard aggregate post-merge: **7.5–8.0** (not 8.5+) unless Semgrep detection lands upstream or we switch SAST tools.

## §2 Pinning strategy

**Tool choice: `pin-github-action` (npm).** [VERIFIED: npm registry — `npm view pin-github-action version` returns `2.x` as of mid-2026; actively maintained by mheap.] Rationale:

- Single-purpose: rewrites `uses: org/repo@vX.Y.Z` → `uses: org/repo@<40-char-sha> # vX.Y.Z` deterministically.
- Pure Node — no Lua impact, no shipped-artifact impact.
- Idempotent on already-pinned references (re-runs are no-ops).
- Open-source MIT.
- Alternatives: `ratchet` (Go binary, cross-CI; viable if Yves prefers polyglot tooling), `frizbee` (Stacklok, also Go, broader scope including container images — overkill for this repo). `pin-github-action` wins on minimum-friction. [CITED: https://github.com/mheap/pin-github-action]

[ASSUMED] StepSecurity's hosted Workflow Generator is a UI alternative, useful for one-shot pinning runs but not for ongoing automation. Not recommended for the deterministic-in-CI pattern Phase 6.1 needs.

**Dependabot SHA-pinning behavior (verified 2026-06):**

- Dependabot has supported updating the human-readable `# vX.Y.Z` comment on SHA-pinned actions **since October 2022** [CITED: https://github.blog/changelog/2022-10-31-dependabot-now-updates-comments-in-github-actions-workflows-referencing-action-versions/]
- Behavior **extended October 2025**: now also corrects the comment when it's stale or incorrect (previously only updated when both SHA and tag changed). [CITED: dependabot-core issue #4691 / #7912 resolution thread]
- No additional `.github/dependabot.yml` config needed beyond what's already shipped. Dependabot opens one PR per outdated SHA, with the new SHA + bumped `# vX.Y.Z` comment in lockstep.
- **Confidence: HIGH.**

### Workflow shape (per D-33, refined)

```yaml
# .github/workflows/sast.yml
name: SAST
on:
  push:
    branches: ["main", "phase-*/**", "phase-*"]
  pull_request:
    branches: ["**"]

permissions: read-all

jobs:
  semgrep:
    name: Semgrep SAST
    runs-on: ubuntu-24.04
    permissions:
      security-events: write   # SARIF upload
      actions: read
      contents: read
    container:
      image: semgrep/semgrep@sha256:<DIGEST>  # pin by digest, not tag
    steps:
      - uses: actions/checkout@<sha> # v7.0.0
        with:
          fetch-depth: 0
          persist-credentials: false
      - name: Run Semgrep
        run: |
          semgrep ci \
            --config p/security-audit \
            --config p/secrets \
            --error \
            --sarif --output=semgrep.sarif
      - name: Upload SARIF
        uses: github/codeql-action/upload-sarif@<sha> # v4.0.0
        with:
          sarif_file: semgrep.sarif
```

Notes on the workflow:

- `container: image: semgrep/semgrep@sha256:<digest>` — pin the container by digest (not tag) to match the SEC-06 pin-everything posture. Pull the digest at planning time via `docker manifest inspect semgrep/semgrep:latest` (Stacklok/Frizbee suggests this pattern for container references) OR fall back to running semgrep CLI in the runner (without container) which lets the SHA-pin rule apply only to the action.
- `semgrep ci --error` — non-zero exit on any ERROR-severity finding (D-33).
- Container approach avoids needing to install Python/Semgrep on the runner. Alternative: `pip install semgrep==<pinned-version>` in a non-container job — slightly slower, but easier to reason about than digest-pinning a container.
- **Recommendation:** use the non-container variant first; it produces deterministic CI runs without container-registry roundtrips.

### Expected first-run findings (dry-run mental model against current `src/*.lua`)

The codebase is 12 small modules (auth, entry, errors, finance, http, i18n, log, mapping, model, pagination, purchases, webbanking_header). The dominant patterns are pure-Lua mappers, HTTP shims around MoneyMoney's `Connection()`, and JSON shape transforms.

Likely Semgrep ERROR-severity findings:

- **`p/secrets`** — **zero predicted ERROR findings.** The codebase doesn't contain literal keys; the test fixtures live in `spec/fixtures/` and are JWT-shaped but `.gitleaksignore`'d. Semgrep's `p/secrets` is regex-based and may surface the fixtures — recommend either (a) keeping `spec/` out of Semgrep's scan via `--exclude spec/`, or (b) suppressing with `-- nosemgrep:generic.secrets...` inline in the fixture loaders. Plan should include a dry-run task to enumerate.
- **`p/security-audit`** — **likely zero to two findings.** The patterns in this ruleset (command injection, unsafe deserialization, SSRF) don't have strong analogues in a pure Lua API wrapper. The most plausible hit is on `tostring(...)` interpolation in `log.lua` or string concatenation in URL building (which we already gate via egress allowlist). If any hits, suppress inline with rationale.

**Planner directive (D-33 caveat):** sequence the SAST PR so the workflow lands **after** any necessary inline suppressions are committed. Recommended order: (a) dry-run task at top of plan (`semgrep ci --config p/security-audit --config p/secrets --error --dry-run` locally), (b) commit any suppressions, (c) commit the workflow YAML, (d) confirm green run, (e) add to required-status-checks.

### Suppression syntax for Lua

[CITED: semgrep docs — ignoring findings] Lua supports two suppression patterns:

- **Inline (preferred):** `-- nosemgrep:<rule-id> rationale here`
- **Next-line:** `-- nosemgrep:<rule-id>` on the line preceding the offending code

Examples:

```lua
-- nosemgrep:generic.secrets.security.detected-jwt-token: test fixture, JWT-shaped, no real claims
local FAKE_JWT_FOR_TESTS = "eyJhbGciOiJIUzI1NiJ9..."

local response = http.post(url, body)  -- nosemgrep:lua.lang.security.audit.SSRF-like — host comes from allowlisted const
```

`--` is Lua's comment marker; Semgrep recognizes both `--` and `#` style depending on the language. Verified for Lua. **Confidence: MEDIUM.** (the canonical Semgrep docs primarily document `# nosemgrep` for Python; Lua follows the language-comment convention — confirm in the dry-run task).

## §8 Egress allowlist hardening (G-01)

Current allowlist grep in ci.yml (lines 87–119) has two gaps the R1 review flagged:

1. **TLD list is incomplete and case-sensitive.** The TLD pattern `(com|net|org|io|dev|app|cloud|sh|de|info|co|biz|me|xyz|tech)` misses common exfil TLDs (`top`, `xyz` covered but `icu`, `monster`, `bet`, `online`, `site`, `space` missing) and is anchored to lowercase only. An attacker who introduces a literal `evil.com` in source would be caught — but a literal `evil.INFO` or `evil.icu` would not.
2. **IP-literal exfiltration bypasses.** Hardcoded IPs like `192.0.2.1` or `[2001:db8::1]` don't match the TLD pattern at all.

**Refactored approach: invert the matcher to an explicit allowlist + reject everything else.**

```bash
- name: Egress allowlist gate — only zettle/izettle hosts in artifact
  run: |
    set -e
    # Defense in depth: enumerate every URL-shaped, host-shaped, and
    # IP-shaped token in the artifact. Reject any that isn't in the
    # explicit allowlist.
    ALLOWED='^(oauth\.zettle\.com|purchase\.izettle\.com|finance\.izettle\.com)$'

    # 1. Scheme-prefixed URLs.
    BAD_URLS=$(grep -Eoi 'https?://[^"'"'"' ]+' dist/paypal-pos.lua \
      | sed -E 's|^https?://||I' \
      | sed -E 's|[/:?#].*$||' \
      | sort -u \
      | grep -Eiv "$ALLOWED" \
      || true)
    if [ -n "$BAD_URLS" ]; then
      echo "FAIL: unexpected scheme-prefixed hosts:"
      echo "$BAD_URLS"
      exit 1
    fi

    # 2. Scheme-less hostnames matching a public-suffix-like shape.
    #    Case-insensitive grep; expanded TLD list including known
    #    exfil TLDs.
    BAD_BARE=$(grep -Eoi '\b([a-z0-9]([a-z0-9-]{0,61}[a-z0-9])?\.)+(com|net|org|io|dev|app|cloud|sh|de|info|co|biz|me|xyz|tech|icu|top|monster|bet|online|site|space|click|link|live|fun|store|loan|date|men)\b' dist/paypal-pos.lua \
      | sort -u \
      | grep -Eiv "$ALLOWED" \
      | grep -Ev '^(account|transaction|credential|error|purpose|http|finance|mapping|auth|entry|payment|payout|fee|sale|refund)\.' \
      || true)
    if [ -n "$BAD_BARE" ]; then
      echo "FAIL: unexpected scheme-less hostnames (S-05 SEC):"
      echo "$BAD_BARE"
      exit 1
    fi

    # 3. IPv4 literals (excluding loopback + RFC1918 which are inert).
    BAD_IPV4=$(grep -Eo '\b((25[0-5]|2[0-4][0-9]|[01]?[0-9][0-9]?)\.){3}(25[0-5]|2[0-4][0-9]|[01]?[0-9][0-9]?)\b' dist/paypal-pos.lua \
      | sort -u \
      | grep -Ev '^(127\.|10\.|192\.168\.|172\.(1[6-9]|2[0-9]|3[01])\.)' \
      || true)
    if [ -n "$BAD_IPV4" ]; then
      echo "FAIL: unexpected IPv4 literal in artifact:"
      echo "$BAD_IPV4"
      exit 1
    fi

    # 4. IPv6 literals (any).
    BAD_IPV6=$(grep -Eo '\[[0-9a-f:]+\]|([0-9a-f]{1,4}:){7}[0-9a-f]{1,4}' dist/paypal-pos.lua \
      || true)
    if [ -n "$BAD_IPV6" ]; then
      echo "FAIL: unexpected IPv6 literal in artifact:"
      echo "$BAD_IPV6"
      exit 1
    fi

    echo "OK: only allowlisted hosts in dist/paypal-pos.lua (scheme + scheme-less + no IP literals)"
```

Key changes:

- `-i` flag on grep for case-insensitive matching (catches `evil.INFO`).
- TLD list expanded with common exfil-favorite TLDs.
- New checks 3 and 4 catch IPv4 and IPv6 literals, with RFC1918/loopback excluded as inert.
- Existing namespace-prefix exclude list expanded to cover new Lua module prefixes added in Phases 3–5 (payment, payout, fee, sale, refund).

**Trade-off:** the IP-literal grep will fire on any literal IP in source. Verified: no IP literals in current `src/*.lua` (confirmed via `grep -rEn '([0-9]+\.){3}[0-9]+' src/` → zero non-comment hits). Safe to add. If a future legitimate IP literal is needed (unlikely), add it to a small allowlist in the script.

## Common Pitfalls

### Pitfall 1: SHA-pinning the wrong target
**What goes wrong:** an action's sub-path (e.g. `github/codeql-action/upload-sarif@<sha>`) requires the SHA of the *parent* `github/codeql-action` repo, not the sub-path. Pinning to a sub-path-specific SHA breaks at workflow load.
**How to avoid:** `pin-github-action` handles this correctly. Manual pinning: always resolve the SHA via `gh api repos/<owner>/<repo>/git/ref/tags/<tag>` against the parent repo only.

### Pitfall 2: Branch-protection lockout
**What goes wrong:** adding a not-yet-existing or never-green status check to required-checks blocks all PRs (including the one trying to remove the broken check).
**How to avoid:** add Semgrep + Scorecard to required-checks **only after** they've each produced one green run on `main`. PR-2 commits the workflow; PR-3 commits the required-checks update.

### Pitfall 3: Dependabot SHA-pin update strategy collision
**What goes wrong:** Dependabot opens an SHA-bump PR that conflicts with manual SHA edits (e.g. if a planner runs `pin-github-action` after Dependabot already opened a PR).
**How to avoid:** close the open Dependabot PRs #2/#3/#4/#12 as part of PR-1's commit (the PR-1 commit superseeds the bumps).

### Pitfall 4: `persist-credentials: false` + a step that needs git push
**What goes wrong:** if a downstream step needs `git push` (none currently — `badge` job does its own `git init`), it silently fails because credentials are gone.
**How to avoid:** verified no step needs push credentials. The `badge` job's `mktemp -d && git init` pattern injects credentials via `https://x-access-token:${GITHUB_TOKEN}@github.com/...` which works regardless of `persist-credentials`.

### Pitfall 5: Top-level `permissions:` blanket-write on release.yml because of the publish job
**What goes wrong:** setting `permissions: write-all` at top-level to cover the publish job's write needs leaks the write scope to verify-signed-tag + build-test-coverage-repro.
**How to avoid:** verified — release.yml already uses the correct pattern (top-level `contents: read`, publish job-level `contents: write`). Phase 6 got this right; Phase 6.1 doesn't regress.

### Pitfall 6: Scorecard aggregate score is slow to update
**What goes wrong:** the `scorecard.dev/viewer/?uri=...` UI caches scores ~24 h. The `api.securityscorecards.dev/projects/...` API is closer to real-time but still has minutes of delay.
**How to avoid:** trigger a manual scorecard.yml run via `gh workflow run` after each PR merges to `main`. Re-query the API a few minutes later.

### Pitfall 7: Semgrep container image pull rate-limit
**What goes wrong:** if pinning Semgrep via a container image (`container: image: semgrep/semgrep@sha256:...`), the registry pulls count against Docker Hub anonymous-rate-limit. Random CI runs may fail with 429.
**How to avoid:** prefer the pip-install path (recommended in §5). If using container, configure `containers:` with `credentials:` against ghcr.io's Semgrep mirror.

### Pitfall 8: Scorecard re-evaluates Branch-Protection on `branch_protection_rule` event
**What goes wrong:** scorecard.yml triggers on `branch_protection_rule` — useful, but each change to branch protection re-evaluates. If we tweak protection without the SCORECARD_READ_TOKEN provisioned, the check returns -1 (unreadable).
**How to avoid:** provision SCORECARD_READ_TOKEN **before** the `setup-branch-protection.sh` re-run in PR-3.
