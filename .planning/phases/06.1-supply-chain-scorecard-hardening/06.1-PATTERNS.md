- **Header style:** YAML `name:` first line (matches scorecard.yml:1, commit-lint.yml:1, ci.yml:1, release.yml:1). No license header — workflow files are infrastructure, not shipped code.
- **Phase-ref convention:** Top-of-file comment block referencing the SEC-ID (e.g. `# Phase 6.1 / SEC-08 — Semgrep SAST as required check.`) — matches release.yml:3–16 and commit-lint.yml:3–9.

## §3 Anti-pattern alerts

The following patterns exist in the codebase and **must not be replicated** in new files / left intact during modifications:

1. **Floating-tag `uses:` references** — every workflow currently uses `@v4`, `@v13`, `@v6.1.0`, `@v2.4.0`, `@v3`, `@v2` (ci.yml:23,28,33,244,248; scorecard.yml:33,38,48,55; release.yml:41,109,114,119,228,246,252; commit-lint.yml:23). This is the entire **SEC-06 work surface**. Every one must be replaced with 40-char SHA + `# vX.Y.Z` trailing comment. Doing one workflow and forgetting another silently drops the Pinned-Dependencies check back to a fractional score.

2. **Sub-shell `awk | sed | cut` parsing of `luacov.report.out` totals** (ci.yml:55–56, release.yml:139). Three piped processes for one numeric extraction; fragile when `luacov` ever changes its `Total` line format. Replace with a single `awk` invocation (e.g. `awk '/^Total/ {gsub("%",""); print int($NF)}'`). Do this in **both** ci.yml and release.yml in the same PR — drifting them is worse than the current state.

3. **Inline 60-line shell scripts inside `run:` blocks** (ci.yml:48–69 coverage gate, ci.yml:83–120 egress gate, ci.yml:122–144 print-gate, release.yml:62–99 verify-signed-tag). Acceptable when load-bearing security gates need to be readable in one place, but **do not extend** these blocks during 6.1. If a new gate is needed, factor it into `tools/ci-<gate>.sh` and call the script. Adding more inline shell to existing inline shell compounds the maintenance debt.

4. **Tag-vs-SHA mixing inside one workflow** — after pinning, a contributor who adds a new step with `@vX` floating tag must be caught. Add a CI step (or extend gitleaks/luacheck-equivalent) that greps `\.github/workflows/.*\.yml` for `uses:.*@v[0-9]` without a following 40-hex-char SHA and fails. This is the corollary gate for SEC-06.

5. **Sprint-proposal's `container: returntocorp/semgrep` skeleton** (proposal line 155) — the image is pinned by tag, not SHA, which contradicts SEC-06. Use the `semgrep/semgrep-action` GHA pinned to commit SHA instead, or pin the container image by digest (`@sha256:...`).

6. **Missing `persist-credentials: false` on every `actions/checkout`** — currently only scorecard.yml:35 sets it. ci.yml:23, ci.yml:244, release.yml:41, release.yml:109, commit-lint.yml:23 all default to `persist-credentials: true`, which leaves an authenticated git token in `.git/config` for any subsequent step to exfiltrate. Add `with: persist-credentials: false` to **every** checkout that does not need to push (i.e. every checkout except the badge job's mktemp-staged push at ci.yml:213–224, which uses an explicit `x-access-token` URL anyway and never touches the working-tree `.git/config`).

7. **(Possible) `name:` string drift between job declarations and `tools/setup-branch-protection.sh:50–54`** — the CHECKS array is byte-identical-coupled to job `name:` declarations. Adding `Scorecard analysis` + `Semgrep SAST` to the array without verifying the exact strings in scorecard.yml:22 and the new sast.yml will silently mis-configure branch protection (the wrong context names produce a "waiting for status check that never arrives" deadlock on every PR).
