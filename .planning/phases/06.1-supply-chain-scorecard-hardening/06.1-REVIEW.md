## Warnings

### P6.1-R-01: MkDocs `--strict` build likely fails for EN locale (missing translations)

**Files:** `mkdocs.yml:40-65`, `docs/en/` (only `index.md` + `readme.md` exist)
**Severity:** WARNING
**Problem:** `mkdocs.yml` configures `mkdocs-static-i18n` with `docs_structure: folder` and `build: true` for both DE (default) and EN. The DE root has `index.md`, `readme.md`, `security.md`, `contributing.md`, `changelog.md`, and `adr/0001..0009-*.md`. The EN folder (`docs/en/`) only ships `index.md` + `readme.md` — there is NO `docs/en/security.md`, `docs/en/contributing.md`, `docs/en/changelog.md`, `docs/en/adr/*.md`. Combined with `mkdocs build --strict` in `.github/workflows/docs.yml:59`, every missing-translation warning surfaced by the i18n plugin escalates to a fatal error. The `validation.links.{not_found,absolute_links,unrecognized_links}: info` block in `mkdocs.yml:72-76` only relaxes link validation; it does NOT silence the i18n plugin's `WARNING - [i18n] Translation file not found for ...` warnings.

Result: the first time docs.yml runs against `main` it likely fails the build, blocking the GitHub-Pages deploy.

**Fix:** pick one of three:
1. **Drop `--strict`** from `mkdocs build` in `docs.yml:59` for the deploy job. Loses validation strictness but unblocks deploy.
2. **Set `build: false` for `en`** in `mkdocs.yml:48`, shipping a DE-only site until EN translations are written. (`nav_translations` in lines 50-65 has no effect when `build: false`.)
3. **Ship EN include wrappers** at `docs/en/security.md`, `docs/en/contributing.md`, `docs/en/changelog.md`, plus `docs/en/adr/*.md` files. The repo-root `SECURITY.md`, `CONTRIBUTING.md`, `CHANGELOG.md` are bilingual; ADRs are English-only — so EN-include wrappers are content-equivalent to DE for ADRs and parallel-content for the rest. Lowest-disruption permanent fix.

Recommended: option 3, mirroring the existing `docs/readme.md` → `../README.de.md` / `docs/en/readme.md` → `../../README.md` pattern.

### P6.1-R-04: `tools/setup-branch-protection.sh` post-condition reads `enabled` fields that may not exist for older GitHub API responses

**Files:** `tools/setup-branch-protection.sh:204-213`
**Severity:** WARNING
**Problem:** The S-R2-L-01 post-condition uses `.allow_force_pushes.enabled // false` and `.allow_deletions.enabled // false`. For a branch where these fields are absent from the API response (e.g. on a freshly-created classic protection rule before the field was populated, or on certain GitHub Enterprise Server versions), the `// false` defaults silently mask a configuration where the field doesn't exist at all. The assertion then passes spuriously. The earlier `enforce_admins` check (line 176) and `required_signatures` check (line 195) have the same shape but those fields are guaranteed present in the classic protection response.

More importantly: the GitHub REST API for classic branch protection returns these two fields ONLY when they have been explicitly set. So `// false` is plausibly safe in the happy path because the script's own PUT payload (lines 80-83) explicitly sets `allow_force_pushes: false` and `allow_deletions: false` before the GET — the fields will be present. But for the manual-fallback path (line 102-131 — "WARNING: gh PAT lacks `Administration: write` scope"), the script exits 0 and the post-condition never runs; if a future refactor inverts the if-arms, the post-condition could pass while the fields are absent, masking a real misconfiguration.

**Fix:** assert presence explicitly:
```bash
if ! echo "${PROTECTION_JSON}" | jq -e '.allow_force_pushes' >/dev/null; then
  echo "FAIL: post-condition: .allow_force_pushes field absent from API response" >&2
  exit 1
fi
ALLOW_FORCE=$(echo "${PROTECTION_JSON}" | jq -r '.allow_force_pushes.enabled')
```
Same for `allow_deletions`. Defense in depth — fail loud on schema drift instead of silently passing.
