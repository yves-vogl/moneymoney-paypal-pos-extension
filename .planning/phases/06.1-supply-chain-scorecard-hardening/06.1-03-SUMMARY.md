# Phase 06.1 Plan 03: Semgrep SAST Workflow Summary

**One-liner:** Shipped `.github/workflows/sast.yml` running `semgrep scan --config p/security-audit --config p/secrets --error` on push + PR with SARIF upload to GitHub code-scanning; local dry-run on `src/**` returned zero findings, no suppressions required. (Rule 1 fix in commit `27d7476`: plan text said `semgrep ci --error`, but `ci` is hard-wired to the Semgrep AppSec Platform login flow and rejects `--error`; switched to `semgrep scan` which is offline-friendly and accepts the flag.)

## What landed

- **New workflow `.github/workflows/sast.yml`** (62 lines). Mirrors `scorecard.yml`'s permissions shape:
  - Top-level `permissions: read-all`
  - Single job `semgrep` with `name: Semgrep SAST` (load-bearing — Plan 06.1-04 will add this exact string to `tools/setup-branch-protection.sh` CHECKS array)
  - Per-job permissions: `security-events: write` + `actions: read` + `contents: read` (minimum needed for SARIF upload)
- **Triggers:** `push` on `[main, phase-*/**, phase-*]` and `pull_request` on `[**]` (matches PATTERNS §2.1)
- **Ruleset:** `p/security-audit` + `p/secrets` only (D-34); no `p/ci`, no Lua-community rules, no `p/owasp-top-ten`
- **ERROR-blocking:** `semgrep scan --error` exits non-zero on ERROR-severity findings (D-33). NOTE: plan text said `semgrep ci --error`; that invocation is invalid (Semgrep CLI 1.167.0 rejects it because `ci` is hard-wired to the Semgrep AppSec Platform login flow and does not accept `--error`). Switched to `semgrep scan` — same rules, same SARIF output, no auth needed (Rule 1 fix).
- **SARIF upload:** `if: always()` so findings appear in the code-scanning UI even when the gate fails
- **Pip install path, NOT container:** `pip install semgrep==1.167.0` avoids Docker Hub rate-limit risk (RESEARCH Pitfall 7); CLI version pinned for deterministic CI

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `semgrep ci --error` is not a valid invocation**

- **Found during:** Task 2 — CI runs `28026173380` and `28026276896` both failed at the "Run Semgrep" step with `semgrep ci: unknown option '--error'`.
- **Issue:** The plan text (and the Task 2 verify gate) codified `semgrep ci --error …` as the canonical invocation. Semgrep CLI 1.167.0 rejects `--error` from the `ci` subcommand because `ci` is hard-wired to the Semgrep AppSec Platform login flow and assumes blocking semantics come from the platform-side ruleset config, not the CLI.
- **Fix:** Switched to `semgrep scan` (the offline-friendly subcommand that accepts `--error` to flip the exit code on ERROR-severity findings). Same rules (`p/security-audit` + `p/secrets`), same SARIF output, same exclude patterns. Added `--metrics=off` for full offline determinism.
- **Files modified:** `.github/workflows/sast.yml`
- **Commit:** `27d7476` (`fix(06.1-03): switch semgrep ci → semgrep scan (--error not supported by ci)`)
- **Verification:** Local re-run with the corrected invocation produced 0 findings + exit 0 + valid SARIF (514 KB). CI run `28026375102` GREEN.

### Adapted-but-not-bug deviations

- **Semgrep CLI version:** Plan suggested `1.96.0` as a placeholder and instructed to re-resolve at execution time. Resolved to `1.167.0` (current PyPI latest stable 1.x) and pinned that in the workflow. Documented as expected per plan text ("or whatever the current 1.x latest is at execution time").
- **Local dry-run command shape:** Used `semgrep scan` (not `semgrep ci`) locally because `semgrep ci` refuses to run without GitHub-Actions context. This pre-saged the Rule 1 fix but at the time was assumed to be a local-vs-CI shape difference. The CI failure revealed `ci` is also non-viable in the workflow itself, hence the Rule 1 fix.
- **Pipx instead of pip:** macOS Homebrew Python enforces PEP 668 (externally-managed-environment), so `pip3 install --user semgrep` is blocked. Used `pipx install semgrep==1.167.0` for the local dry-run; the workflow itself uses `pip install` inside the `actions/setup-python` virtualenv where PEP 668 does not apply. No effect on the workflow's behavior.

No Rule 2 / Rule 3 fixes applied. No Rule 4 architectural changes needed.
