## Bundling decision

The original scope had two natural ways to slice the work: by Scorecard check (Pinned-Deps / Token-Permissions / Branch-Protection / SAST / CII / Docs) and by file ownership (workflows / branch-protection script / `mkdocs` / docs prose). I chose the **file-ownership × Scorecard-check intersection**: each plan owns a non-overlapping `files_modified` set, so the orchestrator can fan out non-conflicting plans in parallel where waves allow.

Two natural collapse points were considered and rejected:

- **Bundling Plan 5 (`.gitleaks.toml`) into Plan 1.** Rejected because `.gitleaks.toml` is a new root-level config and inherits its own anti-pattern review (PATTERNS §1 row 7). Keeping it isolated makes the diff reviewable in one screen and avoids polluting the high-blast-radius SHA-pinning PR.
- **Bundling Plan 6 (Dependabot grouping) into Plan 1.** Rejected because Dependabot needs SHA targets to coalesce against — running it AFTER the SHA-pin PR merges is the only sequence that produces a clean diff (otherwise Dependabot opens 7 stale PRs against floating tags and the planner must close them all by hand).

One natural split was considered and rejected: **splitting Plan 7 (MkDocs site) into "framework setup" + "docs population".** Rejected because the docs site is a single deployable unit; population of the existing Markdown via `nav:` directives is the framework setup (no Markdown is duplicated). A second plan would have nothing to do.

**Net:** 8 plans, average 2 tasks each, every plan ≤ 50 % context.

## Dependency Graph

```
Wave 1 (parallel)
├── 06.1-01 (SHA-pin + perms + persist-creds)        — pre-req for 03, 04, 05, 06
└── 06.1-02 (.gitleaks.toml)                          — independent

Wave 2 (parallel; gated on Wave 1)
├── 06.1-03 (SAST workflow)        ← needs 06.1-01 (SHA targets)
├── 06.1-05 (Dependabot grouping)  ← needs 06.1-01 (SHA targets to coalesce)
└── 06.1-06 (MkDocs site)          ← needs 06.1-01 (SHA-pinned actions in docs.yml)

Wave 3 (sequential after 06.1-03)
└── 06.1-04 (branch-protection)   ← needs 06.1-03 green run on main first
                                    (else required-check deadlock — Pitfall 2)

Wave 4 (gated on Yves bestpractices.dev questionnaire — off-repo CP)
└── 06.1-07 (CII badge)            ← parallel with everything after Yves submits

Wave 5 (consolidation; gated on all above)
└── 06.1-08 (ADR-0009 + SECURITY.md + REQUIREMENTS.md)
```

Sequencing constraints honoured:

1. SHA-pin (01) before SAST (03) — SAST workflow uses SHA-pinned actions from day one.
2. SHA-pin (01) before Dependabot grouping (05) — needs SHA targets to coalesce.
3. SAST (03) before branch-protection (04) — required-status-checks cannot reference a workflow that has not yet produced a green run on `main` (RESEARCH Pitfall 2).
4. Branch-protection (04) requires Yves CP to provision `SCORECARD_READ_TOKEN` secret BEFORE merging.
5. CII badge (07) requires Yves CP to complete bestpractices.dev questionnaire — badge URL is post-questionnaire.
6. MkDocs site (06) is independent of 03/04/05/07; recommended mid-phase so README badges already exist when the docs site goes live, but can ship anywhere after 01.
7. ADR-0009 (08) is the final consolidating doc batch — last.
