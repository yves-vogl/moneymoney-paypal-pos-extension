<objective>
Implement the pure-CPU JWT-payload extraction inside src/auth.lua per D-22: a base64url decoder (RFC 7515 Appendix C translation + MM.base64decode), a JWT payload decoder (split → decode middle → JSON parse), and a `client_id` extractor (`aud` claim with fallback to `client_id` claim). All three are pure-Lua, deterministic, attacker-input-safe (pcall-wrapped JSON parse), and reachable BEFORE any network call so a malformed API key fails synchronously inside InitializeSession2.

Purpose: D-22 closes Phase-1 ADR-0003 Q6 by reading the `client_id` from the assertion's own JWT payload — no shipped partner constant, no second credential field. This plan delivers the load-bearing decoder Phase 2 Wave 3 (entry integration) and the SEC-03 gating test depend on. Without it, InitializeSession2 cannot construct the OAuth form body. The implementation MUST handle every edge case in the RESEARCH.md §"JWT edge cases" table (including malformed padding, two segments, attacker-controlled non-JSON middle) so a hostile or merely-mistyped paste cannot crash the chunk.

Output: ~60 LoC inside the `do … end` block of src/auth.lua attaching `_decode_jwt_payload` and `_extract_client_id` to the pre-declared `M_auth` table (src/webbanking_header.lua L13), plus greened pending tests in spec/auth_spec.lua covering eleven edge cases and four claim-priority cases.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user-pasted API key → _b64url_decode / _decode_jwt_payload | The JWT is OUR credential but the user could paste anything; treat as untrusted input. |
| JSON-parsed JWT payload → _extract_client_id | The payload is a Lua table built from the JWT's middle segment; structurally constrained by `type()` + `#` checks. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Tampering | _decode_jwt_payload with attacker-controlled input | mitigate | JSON parse wrapped in pcall; all type/structure checks before indexing; Lua `error()` cannot escape the function. Spec tests for non-base64url middle, non-JSON middle, missing padding, two segments all assert nil-return without raising. |
| T-02-03-02 | Information Disclosure | API key leakage through log path | mitigate | auth.lua emits ZERO log lines in this plan; only nil-or-table returns. The entry.lua INFO line (Wave 3) does not include the API key. SEC-03 gating test (Wave 4) covers the full path. |
| T-02-03-03 | Spoofing | JWT signature not verified | accept | Per D-22, we read OUR OWN public payload; Zettle's server verifies the signature. RSA primitives are not available in the MoneyMoney sandbox (RESEARCH §"Why no signature verification"). Risk bounded — a fake `aud` only causes a 401 from oauth.zettle.com that we map to LoginFailed. |
| T-02-03-04 | Tampering | base64url alphabet edge cases (e.g., embedded null bytes after decode) | mitigate | Decoded bytes pass through pcall'd JSON parse; non-JSON / structurally-invalid output returns nil. Null bytes inside JSON values are preserved by dkjson and harmlessly compared via `type(s) == "string" and #s > 0` (#-of-string in Lua handles embedded nulls). |
| T-02-03-SC | Tampering | npm/luarocks installs | accept | Zero installs in this plan — pure Lua source edits only. |
</threat_model>
