<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Semgrep rulesets (`p/security-audit`, `p/secrets`) → repo source | Semgrep pulls rulesets at runtime from `semgrep.dev`. A compromised registry could inject rules that suppress real findings. Mitigation: SHA-pinned Semgrep CLI version + ERROR-blocking gate; an attacker would need to also suppress the gate. |
| `pip install semgrep` step → CI runtime | A compromised PyPI package could exfiltrate secrets. Mitigation: pinned version (`semgrep==1.96.0`); Dependabot tracks Python ecosystem if we add `pip` to dependabot.yml (out of scope for this plan; Plan 06.1-05 may extend). |
| SARIF upload → GitHub code-scanning UI | SARIF schema is JSON; malicious findings could attempt SQL-injection-style fingerprint payloads. Mitigation: GitHub's code-scanning API validates SARIF schema; finding details surface in UI but cannot execute. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06.1-03-01 | Tampering | Compromised Semgrep CLI via PyPI | mitigate | Pin `semgrep==1.96.0` (specific version). Re-resolve before commit. Track in backlog for Plan 06.1-05 follow-up (extend Dependabot to `pip` ecosystem). |
| T-06.1-03-02 | Tampering | Suppression rule abuse (`-- nosemgrep:` blanket-ignore) | mitigate | D-34 + plan text forbid blanket-ignore. Per-finding suppressions require a one-line rationale. Code review surfaces any over-broad suppression. |

<pitfalls>
- **Pitfall 1 — Required-status-check deadlock (RESEARCH Pitfall 2):** Do NOT add `Semgrep SAST` to `tools/setup-branch-protection.sh` CHECKS array in this plan. Plan 06.1-04 owns that edit, and only after this workflow has produced one green run on `main` post-merge. Adding it now would deadlock all PRs.
- **Pitfall 2 — `nosemgrep` syntax for Lua (RESEARCH Assumption A3 MEDIUM):** Semgrep's canonical docs primarily document `# nosemgrep` for Python. For Lua, the convention is `-- nosemgrep:<rule-id>`. If a suppression does not take effect, fall back to `# nosemgrep:` (some Lua-aware rules accept it via the lang-agnostic comment matcher).
- **Pitfall 3 — Blanket-ignore via `.semgrepignore`:** Forbidden per D-34. Per-finding inline suppressions only.
- **Pitfall 4 — Docker Hub rate-limit (RESEARCH Pitfall 7):** Do NOT use `container: image: semgrep/semgrep`. Use `pip install semgrep==<pinned-version>` instead.
- **Pitfall 5 — Job name drift:** Job `name:` MUST be exactly `Semgrep SAST` (matches what gets reported as the check-run context). Plan 06.1-04's branch-protection script will reference this exact string.
- **Pitfall 6 — Suppression on a real bug:** If the dry-run surfaces what looks like a real finding (not a false positive), STOP and surface to Yves. A suppression that papers over a real bug is worse than no suppression.
- **Pitfall 7 — Scorecard score reporting lag:** Yves may observe `SAST.score == 0` after merge and wonder why. Plan 06.1-08's ADR-0009 explains this — but during execution between PR-3 merge and ADR-0009 landing (Plan 06.1-08), surface the gap in the SUMMARY explicitly.
- **Pitfall 8 — `--exclude spec/` vs `--exclude spec`:** Semgrep accepts both shapes; the `--exclude spec` (no trailing slash) form is more portable. Verify the dry-run actually skips `spec/` by checking output line count.
</pitfalls>
