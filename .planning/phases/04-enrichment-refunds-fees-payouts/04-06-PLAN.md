    Title line: `# ADR-0004: Finance API scope requirement and fee-fallback dedup contract`

    Status: ACCEPTED — 2026-06-21 — Phase 4 release polish.

    Context: Phase 4 (v0.2.0) adds Finance API integration on top of Phase 2's authenticated network layer. Two design questions surfaced during planning that warrant durable documentation: (1) the OAuth scope requirement — Phase 2's existing access tokens for purchase.izettle.com may NOT have permission to call finance.izettle.com if the user's API key was minted without READ:FINANCE scope (RESEARCH §1.2). (2) the fee-linkage fallback dedup contract — when Zettle's Finance API returns PAYMENT_FEE records with missing or unresolved originatingTransactionUuid, the extension can either persist "this date has been aggregated" state in LocalStorage (Option A, bulletproof against double-booking but amends D-59 "no extension-owned state file") OR cluster fees per-refresh and emit aggregate transactionCode based on Berlin-local date (Option B, simpler but risks double-booking if Zettle later back-fills linkage and a subsequent refresh sees the same date fully linked).

    Decision: Subsection per concern.

    Concern 1 — OAuth scope:
    - Recommended user-facing setup: mint API keys at https://my.zettle.com/apps/api-keys with BOTH scopes READ:PURCHASE AND READ:FINANCE.
    - Existing v0.1.0 users whose key was minted with READ:PURCHASE only must re-mint with both scopes before upgrading to v0.2.0.
    - The extension does NOT detect the scope mismatch explicitly — a 401 on the first Finance API call surfaces as the German LoginFailed string (Phase 2 behavior). README v0.2.0 documents this so users diagnose correctly.
    - A future Phase 5/6 may add a scope-specific German error string to distinguish this case from genuine credential invalidity.

    Concern 2 — fee-fallback dedup (Option B chosen for v0.2.0):
    - Per-refresh date clustering: when ANY fee on a given Berlin-local date has unresolved linkage (originatingTransactionUuid does not map to a payments_by_uuid entry in the current refresh), ALL fees on that date are aggregated into a single transactionCode `zettle:fee:aggregate:<YYYY-MM-DD>`.
    - When ALL fees on a date have resolved linkage, per-sale `zettle:fee:<originatingTransactionUuid>` rows emit instead.
    - Trade-off accepted: if a date is aggregated in refresh N and the SAME date in refresh N+M sees all fees newly-linked, the extension emits per-sale rows on refresh N+M while the aggregate row from refresh N remains in MoneyMoney's database — double-booking. README explicitly documents this and instructs users to manually edit (delete) the stale aggregate row if it occurs.
    - This decision amends D-59 "no extension-owned state file" by accepting the double-booking risk in exchange for keeping the extension stateless; Phase 5+ may revisit if real users hit the failure mode.

    Concern 3 — temporal payout inference:
    - Zettle's Finance API does not publish a documented PAYOUT-to-PAYMENT link field (RESEARCH §4.1). The extension infers settlement by selecting the earliest PAYOUT whose timestamp is greater-than-or-equal-to the PAYMENT's timestamp.
    - Conservative-miss behavior: a PAYMENT made today with no PAYOUT yet visible stays booked=false until a future refresh sees the PAYOUT. No false positives.
    - Edge case: monthly-payout merchants may experience 1-2 refresh-cycle delay before sales are promoted to booked=true. README documents this.
    - FROZEN_FUNDS / ADJUSTMENT carve-outs between PAYMENT and PAYOUT are not modeled — out of Phase-4 filter scope.

    Consequences:
    - Positive: extension surfaces full bookkeeping picture (settled + pending balance, refunds linked, fees per-sale or aggregate, payouts as negatives) without persistent state.
    - Negative: scope-upgrade required for existing users; aggregate-may-double-book risk; some payouts may be inferred conservatively (sale stays booked=false 1-2 cycles longer than ideal for monthly-payout merchants).
    - Mitigations: README upgrade-path section + Bekannte Grenzen section; Phase 5 may revisit fee dedup with persistence and scope-specific error strings.
