# Phase 6.1: Supply-chain & Scorecard hardening - Context

**Gathered:** 2026-06-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Lift the project's OpenSSF Scorecard aggregate from the current 5.2 / 10 to **≥ 8.5 / 10** on `main` HEAD, without compromising the solo-maintainer constraint, by landing five hardening PRs (pinned actions + Dependabot SHA bumps, workflow-token least-privilege, branch-protection-enforced-and-introspectable, Semgrep SAST as a blocking required check, CII Best Practices "passing" badge) plus one documentation PR (ADR-0004 + SECURITY.md "Supply-chain controls" section). The supply-chain posture becomes a published, auditable prerequisite for any future v1.0.0 release.

**In scope:**
- `.github/workflows/ci.yml` refactored into two jobs (`test` read-only + `badge` write-scoped, only on push to main)
- `.github/workflows/scorecard.yml` rewired with `SCORECARD_READ_TOKEN` for Branch-Protection introspection
- `.github/workflows/sast.yml` (new): Semgrep on push + pull_request, SARIF → code-scanning, `semgrep ci` blocking on findings
- All `uses:` references across the three workflows pinned to commit SHA + `# vX.Y.Z` comment
- `.github/dependabot.yml` extended to bump action SHAs and preserve the version-tag comment
- Branch-protection rules on `main` set via GitHub Settings (off-repo action, scripted-reproducibly-documentable in `docs/adr/0004`)
- CII Best Practices `passing` self-assessment filled in, badge URL rendered in `README.md` (German README primary) and `README.de.md` (or its eventual successor)
- `docs/adr/0004-openssf-scorecard-stance.md` documenting the deliberately accepted gaps (Fuzzing, Solo Code-Review, Packaging, Contributors, Maintained)
- `SECURITY.md` extended with a "Supply-chain controls" section enumerating the active mitigations (pinned actions, SAST, signed releases [scaffolded in Phase 6], redact-before-log, egress allowlist)
- `.planning/REQUIREMENTS.md` extended with the new IDs locked below (D-36)

**Out of scope (explicitly delegated to Phase 6):**
- Sigstore/cosign signing of the release artifact (Plan 6.5-06 in the proposal explicitly belongs to Phase 6 — see D-30)
- SLSA-Provenance Level 3 generator (same reason — Phase 6)
- GPG-signed tag-trigger workflow (Phase 6 — release pipeline)
- The Signed-Releases Scorecard check will resolve when Phase 6 ships; Phase 6.1 does not target it
- BUILD-01/BUILD-02 wording in REQUIREMENTS.md is NOT modified in 6.1 — the roadmap's "anticipated update" of these two IDs (Sigstore/cosign, SLSA-Provenance) lands with Phase 6 instead

**Out of scope (permanently, per accepted-gap policy):**
- Fuzzing (Lua wrapper, no parser/codec — Scorecard Fuzzing stays at 0/10, documented in ADR-0004)
- Bot-driven Code-Review (CodeRabbit/Greptile/Diamond — Code-Review stays at 0/10 while solo-maintainer)

**Waits-itself-out:**

<decisions>
## Implementation Decisions

### Sigstore / SLSA — out of 6.1 (Area A)
- **D-30:** Sigstore/cosign signing of `dist/paypal-pos.lua` and SLSA-Provenance Level 3 are **deferred entirely to Phase 6**, per Plan 6.5-06 of the sprint proposal. Phase 6.1 does NOT touch `BUILD-01` or `BUILD-02` in REQUIREMENTS.md. Rationale: signed-releases hardening needs the Phase 6 tag-triggered release pipeline as a substrate; splitting it earlier doubles the integration surface for no Scorecard win (Scorecard's Signed-Releases check only fires on actual releases, which Phase 6.1 does not produce).

### Branch-protection on `main` (Area B-1)
- **D-31:** Branch-protection rules on `main` are set to the **maximum-strict** profile, no admin bypass:
  - Require pull request before merging (1 approval — auto-met when the maintainer self-approves via a separate-session review, or via a sanctioned bot if one is later introduced; for now Scorecard Code-Review check stays at 0/10 as an accepted gap)
  - Require **signed commits** (aligns with Phase-1 D-13: all commits already GPG-signed under key `FDE07046A6178E89ADB57FD3DE300C53D8E18642`)
  - Require **linear history** (no merge commits)
  - Require status checks to pass: **CI (test + badge)**, **Scorecard analysis**, **Semgrep SAST**
  - **Force-push blocked**, **branch-delete blocked**
  - **No admin bypass** — Scorecard penalizes admin-bypass with ≤0.5 points; the solo maintainer accepts living with the same rules as any external contributor
  - The `coverage-badge` branch is an unprotected orphan branch and is exempt from these rules (it's force-pushed by CI; see D-32)

### Permissions refactor of `ci.yml` (Area B-2)
- **D-32:** `.github/workflows/ci.yml` is refactored into **two jobs**:
  - Top-level: `permissions: read-all`
  - Job `test` (PR + push trigger): `permissions: { contents: read }`; runs luacheck + busted + luacov + `tools/build.lua` + `tools/build.lua --verify` + the four existing gates (DEBUG=false, egress-allowlist, no-AI-attribution, build-verify)
  - Job `badge` (`needs: test`, `if: github.event_name == 'push' && github.ref == 'refs/heads/main'`): `permissions: { contents: write }`; runs the coverage-badge generation and force-push to the `coverage-badge` orphan branch
  - **All four existing CI gates are preserved verbatim** in the `test` job (DEBUG=false grep, egress-allowlist grep, build --verify, no-AI-attribution grep)
  - The `coverage-badge` force-push survives signed-commits-on-`main` (D-31) because `coverage-badge` is a different branch and is not under main's protection rules

### SAST gate (Area C-1)
- **D-33:** Semgrep findings are **blocking** for merges to `main`. New workflow `.github/workflows/sast.yml`:
  - Triggers: `push`, `pull_request`
  - Top-level `permissions: read-all`; the single job declares `permissions: { security-events: write, actions: read, contents: read }` for SARIF upload
  - Command: `semgrep ci --error` (non-zero exit on any ERROR-severity finding)
  - SARIF output uploaded to GitHub code-scanning via `github/codeql-action/upload-sarif@<SHA>`
  - This job is added to the required-status-checks set in branch-protection (D-31)
  - **Caveat:** the first PR to enable this may surface unexpected ERROR-level findings against existing `src/` code; planner should sequence SAST-enable **after** any necessary suppressions are committed (use `# nosemgrep:` inline comments with rule ID + rationale, never blanket-ignored)

### SAST ruleset (Area C-2)
- **D-34:** Semgrep ruleset for v1 is **`p/security-audit` + `p/secrets`** only. Rationale:
  - `p/security-audit`: generic security patterns (command injection, SSRF-like patterns, unsafe deserialization) — bites every supported language including Lua's small ruleset
  - `p/secrets`: API-key / token / private-key leak patterns — directly reinforces SEC-01 redaction posture
  - Lua-specific community rules (`semgrep.dev/r?lang=lua`) are **not** included in v1 (the available set is ~15 rules, most low-signal for an API-wrapper extension); revisit if Phase-2/3 code triggers a real false-negative
  - `p/ci` (GitHub-Actions hardening) is NOT included in v1 either; Pinned-Actions + Token-Permissions hardening already covers the same threat model via Scorecard checks

### CII Best Practices badge target (Area D-1)
- **D-35:** Phase 6.1 acceptance is **CII "passing" level (≥5/10)**, badge URL rendered in `README.md`. Silver (~7/10, +3 h work, +0.3 Scorecard aggregate) is **tracked as a backlog issue**, not gated on for 6.1. Rationale: roadmap success criteria #6 says "passing badge" verbatim — passing satisfies the acceptance bar; silver is incremental value, not 6.1-critical. Phase 6.1 ships with the passing badge live and an open issue titled "CII Best Practices → Silver" linked from `docs/adr/0004` so it's not lost.

### Requirement IDs (Area D-2)
- **D-36:** The following requirement IDs are **locked here and will be appended to `.planning/REQUIREMENTS.md` as the first task of the plan**. ID collision check: `SEC-05` and `BUILD-03` are **already taken** in REQUIREMENTS.md (SEC-05 = branch-protection signed-commits, BUILD-03 = `__VERSION__` substitution). New IDs therefore start at SEC-06 and BUILD-07; SEC-05 is **expanded in-place**.

  | ID | Status | Acceptance text (source: ROADMAP §Phase 6.1 success criteria) |
  |----|--------|----|
  | **SEC-05** (existing — **expanded**) | Modify | Branch protection on `main` requires GPG-signed commits, linear history, force-push and delete blocked, no admin bypass, AND a fine-grained PAT `SCORECARD_READ_TOKEN` with `Administration:read` scope is configured so the Scorecard Branch-Protection check can introspect; Scorecard `Branch-Protection.score >= 8` post-merge. |
  | **SEC-06** (new) | Add | Every `uses:` reference in `.github/workflows/*.yml` is pinned to a 40-char commit SHA followed by a `# vX.Y.Z` comment; Dependabot is configured to bump the SHA and preserve the tag comment; Scorecard `Pinned-Dependencies.score == 10` post-merge. |

### Claude's Discretion
- Exact split of the five hardening PRs vs one documentation PR vs ordering — planner's call, subject to the constraint that **SAST-enable must follow any needed Semgrep suppressions** (D-33 caveat) and **Token-Permissions refactor must come before or with the SAST workflow** so the SAST workflow can be written with the right permissions shape from the start.

## Existing Code Insights

### Reusable Assets
- **`.github/workflows/ci.yml`** — already implements the four invariant CI gates (`DEBUG = false` grep, egress-allowlist grep on `dist/paypal-pos.lua`, no-AI-attribution grep on the working tree, byte-reproducible build via `tools/build.lua --verify`); these are the load-bearing gates from Phase 1 (SEC-04, D-07, D-12) and **must be carried verbatim into the new `test` job under the D-32 split**.
- **`.github/workflows/scorecard.yml`** — already declares `permissions: read-all` top-level + job-local `security-events: write` + `id-token: write`; this is the template the new `sast.yml` should mirror (D-33, SEC-07).
- **`.github/dependabot.yml`** — already runs `package-ecosystem: github-actions` weekly with grouping disabled; the SHA+comment preservation behavior (SEC-06) is mostly a docstring-and-test concern, not new YAML.

### Established Patterns
- **GPG-signed commits already universal** (Phase-1 D-13) — the signed-commits branch-protection rule (D-31) is free of operational disruption; no need for a "soak period" before enabling it.
- **Egress allowlist is the project's load-bearing security invariant** — any refactor that breaks the egress-grep gate is by definition a regression. The new `test` job in D-32 must keep this grep.
- **No-AI-attribution gate already excludes `.planning/`, `CLAUDE.md`, `.github/`** — Phase 6.1's new `sast.yml` is implicitly covered by the existing `.github/` exclusion; no further changes needed.
- **README.md is German-primary** — CII badge insertion (BUILD-07) goes into `README.md` first; the bilingual `README.de.md` does not yet exist (Phase 6 territory), so badge goes into the current single README until that split lands.

### Integration Points
- **Branch-protection rules are off-repo state** (set in GitHub Settings UI / via `gh api`). The planner must script the change as a documented `gh api` command sequence in `docs/adr/0004` or a `tools/setup-branch-protection.sh` so the configuration is reproducible from the repo (and so the maintainer can re-apply it on a fresh fork).
- **`SCORECARD_READ_TOKEN`** is a fine-grained PAT — its existence and rotation cadence belong in `SECURITY.md`'s new supply-chain section; the planner notes the expiry (≤ 1 year per GitHub policy) and the rotation procedure.
- **CII Best Practices badge URL** depends on a real account at `bestpractices.coreinfrastructure.org`; this is the only off-repo manual step the maintainer must do during execution (CII does not have a fully scriptable API for self-assessment submission).
